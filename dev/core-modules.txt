# Pure-core module allowlist (dev/PLAN.md §0/D5).
#
# The HARD coverage-floor tier: pure, data-in/data-out modules where "cover every branch" and
# "write a real behavior test" are the same act — so a hard floor here is free of the
# test-padding failure mode and is the highest-value place to harden (a wrong SQL-context
# silently corrupts every autocomplete suggestion).
#
# One repo-relative path prefix per line (matches files whose cobertura filename starts with
# it); blank lines and `#` comments ignored. Grows as pure-core modules land per phase.
#
# Phase 1 (present now): schema model + types.
src/schema.rs
src/schema/types.rs
src/engine/types.rs

# Phase 4 (present now): faithful temporal/decimal/blob/interval value rendering — pure
# data-in/data-out conversion of a DuckDB ValueRef to its canonical display text. A wrong civil-date
# or fraction-trim is a real output-corruption bug (it shipped `Date32(19372)` once), so it earns the
# hard floor.
src/engine/value_render.rs

# Phase 2 (present now): SQL preprocess, query_state stale-discard, grid layout/col_width math,
# and the query-bar editor (pure char-indexed buffer + cursor — fully covered by its tests).
src/query/preprocess.rs
src/query/query_state.rs
src/grid/grid_layout.rs
src/grid/col_width.rs
src/app/editor.rs

# Phase 2/5 (present now): DuckDB-error -> friendly-message mapping + the P5.3 did-you-mean (a
# bounded edit-distance over the schema headers). Pure `&str -> String`, one golden row per known
# error category — every branch is a real mapping case, so it earns the hard floor.
src/query/error_enhance.rs

# Phase 5 (present now): polish (P5.3) — the pure empty-state message picker and the truncation
# banner formatter ("showing first N rows" when the grid is ciq-capped). Both data-in/data-out
# (the App's row-count + cap flag in, a string out), so they earn the hard floor.
src/app/polish.rs

# Phase 3 (present now): the shared SQL lexer (D6) — neutral top-level module both `query` and
# `autocomplete` consume; pure, total `tokenize(&str) -> Vec<Token>`.
src/sql_lexer.rs

# Phase 4 (present now): the shared fuzzy-filter primitive — neutral top-level leaf both the
# autocomplete ranker and the column palette consume (lifted out so neither duplicates it nor
# couples to the other); pure, total `is_subsequence(&str, &str) -> bool`.
src/text_match.rs

# Phase 3 (present now): the clause-context detector — pure `detect_context` over the lexer token
# stream; a wrong cursor-context silently corrupts every suggestion, so it earns the hard floor.
src/autocomplete/clause_context.rs

# Phase 3 (present now): the value-source SQL builder + ValueCache — pure string builder (with
# identifier quoting) + plain owned data cache (no engine handle).
src/autocomplete/value_source.rs

# Phase 3 (present now): the candidate generator — pure `get_suggestions` over &Schema +
# &OperatorTable + &ValueCache; a wrong source/branch silently corrupts every suggestion, so it
# earns the hard floor.
src/autocomplete/candidates.rs

# Phase 3 (present now): autocomplete insertion — pure `insert_suggestion` (replace-the-partial +
# SQL identifier/value quoting). Every branch is a real behavior case (which span is replaced,
# when an identifier is quoted, when a value is a string literal), so it earns the hard floor.
src/autocomplete/insertion.rs

# Phase 4 (present now): CSV ingest — the pure sniffer over bytes, the CsvOpts merge precedence +
# read_csv SQL builder, and the shared SQL identifier/literal quoting. All data-in/data-out: a
# wrong delimiter guess, a wrong precedence pick, or a mis-escaped identifier is a real behavior
# bug, so they earn the hard floor.
src/ingest/sniff.rs
src/ingest/csv_opts.rs
src/sql_ident.rs

# Phase 4 (present now): output modes — pure `render_output` (CSV RFC-4180 quoting, TSV escaping,
# JSON null-vs-"" type fidelity, Markdown alignment). Every branch is a real format-correctness
# case (a mis-quoted field corrupts the export; a wrong JSON shape loses the §8/Q12 null-vs-empty
# distinction), so it earns the hard floor.
src/output/emit.rs

# Phase 4 (present now): the column palette generated-state core (D3) — the pure
# toggle/reorder/filter/predicate/ownership state machine, and the pure `emit(state) -> String`
# query builder (both identifier and facet-value quoting surfaces). The emitted byte format is a
# stable identity surface (the ownership byte-compare relies on it) and a mis-quoted identifier or
# value corrupts the generated query, so both earn the hard floor. The `palette_render` blit is
# left off (a TestBackend snapshot seam, like `autocomplete_render`).
src/palette/palette_state.rs
src/palette/query_emit.rs

# Phase 4 (present now): instant facets (P4.6, §6.5) — the pure type-aware facet SQL builder, the
# FacetState result-parse machine, and the pure format_facets line builder + histogram bar-width
# math. A wrong SQL shape, a mis-parsed result, or wrong bar arithmetic silently corrupts the facet
# popup, and all three are data-in/data-out, so they earn the hard floor. The thin render_facet blit
# shares facet_render.rs but is fully exercised by the TestBackend snapshot + zero-area tests, so it
# doesn't erode the floor (same seam treatment as the other *_render blits).
src/facets/facet_query.rs
src/facets/facet_state.rs
src/facets/facet_render.rs

# Phase 5 (present now): the full config schema (Q5) — the pure per-section accessors that fold in
# defaults (row_limit clamp, ai is_active, history max_entries clamp, theme mode/override lookup). A
# wrong default or a wrong is_active gate is a real behavior bug (it silently mis-bounds the
# viewport or silently activates/deactivates the AI feature), and all are data-in/data-out, so they
# earn the hard floor.
src/config/general.rs
src/config/ai_config.rs
src/config/history_config.rs
src/config/theme_config.rs

# Phase 5 (present now): AI NL->SQL (P5.1) — the pure prompt builder (schema-grounded model prompt:
# table name + every column + its ColumnType) and the pure AI popup state machine (editing /
# pending / success / error transitions, UTF-8-safe input). A wrong prompt loses the schema
# grounding the model needs; a wrong popup transition desyncs the request lifecycle — both
# data-in/data-out, so they earn the hard floor. The provider (real HTTP escape hatch + mock), the
# AI thread bridge, and `ai_render` (the TestBackend popup blit) are seams left off the floor
# (the provider's `complete` is the network edge; the bridge is thread+channel plumbing; the blit
# is snapshot-covered like the other *_render files).
src/ai/prompt.rs
src/ai/ai_state.rs

# Phase 5 (present now): query history (P5.2) — the pure ring state machine (add/dedupe-to-newest-
# first/recall/popup-navigate/inline-cycle/fuzzy-filter over an in-memory Vec). A wrong dedupe, a
# wrong navigation clamp, or a wrong filter silently corrupts recall, and all are data-in/data-out,
# so it earns the hard floor. The on-disk `storage.rs` (the only I/O) is a tempdir-tested seam, the
# pure `history_events.rs` key->action map is covered by its tests, and `history_render.rs` is a
# TestBackend blit (same seam treatment as the other *_render blits) — all left off the floor.
src/history/history_state.rs
